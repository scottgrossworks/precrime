## Thread 1 - Procedural conductor (parallelization)

### The gap
There is no real parallelism today. One Goose process is both control panel
and serial executor: it loops claim -> dispatch -> complete one Task at a time.
It stalls, loses claimed-Task context, and invents spurious menus.

### Decisions
- The conductor is a PROCEDURAL Node loop, NOT an LLM. Putting an LLM on top
  recreates the same failure one layer up. The LLM stays a worker brain only.
- The conductor lives INSIDE the `mcp_server.js` process, because that process
  owns the single SQLite write handle plus the Planner (`plan_tasks`) and the
  Judge (`judge_affected`). Co-located means it coordinates by direct function
  call and stays the only DB writer.
- Process ownership inverts the usual Goose stdio convention. The launcher
  (`precrime.bat` / docker entrypoint) starts `mcp_server.js` FIRST, in HTTP
  mode (`StreamableHTTPServerTransport` on `127.0.0.1:5179`). The conductor,
  inside that process, then spawns the Goose workers. Workers do NOT spawn
  their own server; their precrime extension is `type: streamable_http`
  pointed at `127.0.0.1:5179`. One server, N workers, one writer.
- PUSH-claim: the conductor hands each spawned worker exactly one Task. Workers
  have no claim logic and never call `plan_tasks` / `rescore` / `judge_affected`.
- One-shot workers: claim one Task, do it, `complete_task`, exit.
- FAIL-AND-FORGET: a failed Task is garbage-collected. The signal that spawned
  it remains, so a future plan re-covers the ground, or the system moves on.
  No claim-lease, no reaper.

### Rejected as over-engineering
Claim-lease / TTL reaper, epoch / generation fencing, transactional outbox,
staggered spawn ignition (tabled - not a problem we have hit). Rule applied:
never store in SQLite what can be deduced at runtime.

### Module split (refactor `mcp_server.js`)
`mcp_server.js` is one large file doing three jobs. Cut into:
- a DB module: all Prisma CRUD (clients, bookings, factlets, tasks). One place
  anyone reads/writes the sqlite db, including a human inspecting it.
- the conductor: spawn workers + regroup loop. New.
- `mcp_server.js`: shrinks to the HTTP transport + tool wiring that calls the
  other two.
Read `C:\Users\Scott\Desktop\WKG\PRECRIME\server\mcp\mcp_server.js` fully to
map exact boundaries (DB vs conductor vs Planner/Judge vs tool shell) BEFORE
cutting.

### Process tree
```
launcher  (precrime.bat / docker entrypoint)
  +-- node mcp_server.js   HTTP MCP @127.0.0.1:5179
      |                    + conductor + Planner + Judge + the ONLY SQLite writer
      +-- goose worker #1  precrime ext = streamable_http -> 5179   (one Task, exits)
      +-- goose worker #2  ...
      +-- goose worker #N
```

## Thread 2 - Evidence model (why leeds never promote, and the fix)

### The bug (verified, see the solutions doc)
Bookings never reach `leed_ready` because of SAVE-time live-URL re-verification,
not the demand gate. `verifyEvidenceUrl` (~line 1862) and
`verifyResolvedDateSource` (~line 1489), called from `validatePatchEvidenceUrls`
(~line 1888) and the factlet-save path (~line 2808), re-fetch the booking's
`sourceUrl` and reject the save unless that single page literally contains the
year, the exact start month+day, and a proof term. RFP, PDF, and aggregator /
county pages fail, so the row cannot carry the provenance the `leedReady` gate
requires. Separately, the `verification` block in `DOCS/SCORING.json`
(`mustContain`, `rejectIf`, `liveUrlRequired`) is DEAD - no code reads it, so
editing it does nothing. Full diagnosis:
`C:\Users\Scott\Desktop\WKG\PRECRIME\docs\solutions\logic-errors\leed-ready-blocked-by-hardcoded-url-verification.md`

### Decisions
- DELETE the live-URL re-verification from the save and share paths. Capture
  proof ONCE when a worker first reads a page (store the snippet + the date
  seen on the factlet). Trust the captured evidence afterward. No live re-fetch
  on save or share.
- Demand is a human-level judgment, not regex or a count. An LLM reads the whole
  synthesized dossier and writes a STORED verdict. It runs ONLY when the
  client's dossier changes (new factlet, enrichment, contact update); between
  changes the Judge just reads the stored verdict. Trigger is "dossier touched,"
  not the clock. This consciously OVERRIDES FOUNDATION.md's "demand is never
  stored, recomputed every score" - that rule assumed a cheap procedural
  recompute; the recompute is now an LLM call. FOUNDATION.md must be edited.
- Regex / procedural still pulls the hard fields (location, start/end time, zip,
  contact) into the dossier.

### Two independent labels (NOT a ladder)
`leed_ready` and `outreach_ready` are separate criteria on the same booking,
evaluated independently. A booking can be both, one, or neither.
- `outreach_ready` (lower bar) = real direct contact + a dossier rich enough to
  write a quality, specific email.
- `leed_ready` (higher bar) = a clear demand signal + every field the Leedz
  addLeed schema requires (contact, location, zip, date, start/end time, etc.),
  because sharing asks ANOTHER vendor to spend time and money on the client.
Most `leed_ready` are also `outreach_ready` in practice (contact + info already
held), but that is an observation, not a rule. Outreach can be the PATH to share:
email the client, they reply with date/venue, now it clears the addLeed bar; and
if the reply is unattractive (bad date/pay/drive) the user can post it to the
marketplace instead of taking it himself.

Build implication: DROP the current "highest passing gate wins" precedence
(`leed_ready` else `outreach_ready` else `brewing`). It forces one status and
hides the overlap. Compute two independent derived checks at runtime:
- `emailable` = real contact + email-quality dossier
- `shareable` = demand verdict marketplace-viable + full addLeed fields
Both derived at runtime from the stored demand verdict + the regexed fields, not
stored. `booking.status` stays only for lifecycle / terminal
(`brewing`, `shared`, `expired`).

### Demand can be derived, and share-worth is not fee-worth
- DERIVED / inferred demand counts for share, not just explicit requests. The
  Prom Pattern (decision-maker + event + past vendor use + thematic fit) is a
  valid inferred signal even with no literal "we want X." The LLM verdict owns
  this.
- SHARE-WORTHY is not FEE-WORTHY. A leed can be shared for `pr:0` (free) because
  it is worth another vendor's time. Price (including free) is its own field,
  NEVER a gate.
- A missing end time must NOT block the share. `et` becomes a tunable: default
  it (start + a configured duration per event type) or allow share without it.

### Scoring: drop the number as a gate
The numeric 0-100 score and the 60/90 thresholds go away as gates. Promotion is
decided by the two label checks above. Ordering WITHIN a class (which hot leed to
show first) is derived at runtime: soonest event first, then demand strength.
Do not store a composite score.

### Markdown-tunable policy (headline requirement)
Every subtlety above must be TUNABLE IN MARKDOWN that the engine actually reads:
what is hard-required vs derivable, how strong an inferred signal must be to
share, default pricing, end-time defaulting. This is the live tuning surface the
dead `SCORING.json` verification block pretended to be but never was. Policy
lives in editable prose, not hardcoded and not in unwired JSON. Wire it or
delete the dead block.

### "Show me the hot leedz" (ties the threads together)
By the time anyone asks, the expensive work is already done and stored:
enrichment changes a dossier -> the LLM re-judges -> the Judge sets the booking
label. So "hot" is a cheap query over already-judged state, no LLM at ask-time:
```
(shareable OR emailable) AND startDate within now .. now + hotDaysOut
order by startDate asc, demand strength desc
```
- Interactive: the planner creates SHOW_HOT_LEEDZ and the presenter shows that
  sorted list; the user decides share / email / skip per booking, weighing the
  things a machine cannot (money, schedule, drive, desire).
- Headless: precrime is set to ONE objective and acts on it blanket - marketplace
  mode shares the shareable hot ones, outreach mode drafts emails. No per-leed
  taste, because that judgment cannot be automated yet.
Keep `hotDaysOut` at 14 unless changed.
