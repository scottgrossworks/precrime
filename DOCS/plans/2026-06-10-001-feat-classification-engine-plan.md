# feat: PRECRIME cold/brewing/hot classification engine

**Repo:** PRECRIME (source tree at the WKG root). All paths below are PRECRIME-relative.

## Summary

Replace the numeric scoring and gate-based promotion in `server/mcp/mcp_server.js`
with the model in `DOCS/CLASSIFICATION.md`: three states (cold, brewing, hot) stored
in `Booking.status`, no new schema. Promotion to hot is an LLM holistic judgment, not
a points threshold. The two actions on a hot leed (outreach vs marketplace) are chosen
at act time (user in interactive, the configured objective in headless), not stored as
labels. Retire `DOCS/SCORING.json`'s points system to a handful of real knobs, and
delete the live-URL re-verification that currently blocks promotion (the verified
Catch-22). Data is regenerated by rerunning the flow, not migrated.

## Problem Frame

Today `computeBookingTargetScore` (mcp_server.js ~:1075-1164) computes a 0-100
`bookingScore`, evaluates `leedReady` / `outreachReady` gates via `evaluateGate`
(:690) against `DOCS/SCORING.json`, and sets `status` to `leed_ready` /
`outreach_ready` / `brewing`. Two problems:

1. The numeric score is hard to tune and an LLM cannot produce a consistent ordinal,
   so the demand judgment it gates on is unreliable (see origin: DOCS/CLASSIFICATION.md).
2. Promotion is additionally blocked by save-time live-URL re-verification
   (`verifyEvidenceUrl` :1862, `verifyResolvedDateSource` :1489, via
   `validatePatchEvidenceUrls` :1888 and the factlet-save path), which rejects RFP /
   PDF / aggregator pages and creates a Catch-22 where bookings never reach a ready
   state (see DOCS/solutions/logic-errors/leed-ready-blocked-by-hardcoded-url-verification.md).

The classification (cold/brewing/hot) is the only stored verdict. Sharing's harder
requirements (full addLeed contract, precise location, future date) are already
enforced at the addLeed boundary (the Leedz Lambda work completed 2026-06-10), so
PRECRIME does not precompute a marketplace-ready flag.

## Requirements Trace

- States cold/brewing/hot in `Booking.status`, no new column (origin: States section).
- Cold is start and end: a leed returns to cold after it is acted on (outreach drafted
  or shared); it can warm again later.
- Hot = LLM judges a real, fitting, in-demand opportunity. In headless the bar is the
  active mode's; in interactive the lower outreach bar so the user sees candidates.
- Action chosen at act time, not stored: interactive presents hot leedz; headless acts
  per `Config.defaultBookingAction`.
- No numeric score as a promotion gate. Ordering within hot = soonest event first.
- Tuning lives in a minimal `DOCS/SCORING.json` (hotDaysOut, futureMinHours, the
  addLeed required-field list, defaults pr 0 / sh '*').
- `et` and `ph` optional; never synthesized (origin: End time section).
- Delete live-URL re-verification; capture proof once at fetch.

## Key Technical Decisions

- **KTD1: status string carries the classification; no new schema.** `Booking.status`
  already exists. New value set: `cold | brewing | hot`. The old `outreach_ready` /
  `leed_ready` / `taken` / `needs_enrichment` / `new` values are retired; `shared` is
  already tracked by the `Booking.shared` boolean + `sharedAt`, so "acted on" sets
  status back to `cold` rather than a terminal status. Data regenerated, not migrated
  (origin + Thread 3).
- **KTD2: hotness is one LLM call, mode-aware, run on dossier-change.** Replace the
  procedural `detectDemandSignal` + binary `_llmJudgeDemand` with a single judge that
  reads the synthesized dossier + booking and returns `cold | brewing | hot` against
  the active bar. No ordinal, no stored per-label verdict (origin: "no fine
  gradations"). The active bar comes from `Config.defaultBookingAction` in headless;
  interactive uses the outreach (lower) bar.
- **KTD3: the procedural floor is checked at act time, not stored.** A pure JS helper
  answers "can a complete addLeed payload be composed from this dossier?" (required
  fields present, start date in the future). It gates the share action only; addLeed
  itself remains the final judge of location precision and acceptance. Outreach needs
  only a real client + real direct email.
- **KTD4: keep `Client.dossierScore` as an internal enrichment-priority signal only.**
  It is load-bearing for client sorting in `find` and the status report. It is NOT a
  promotion gate and does not appear in booking classification. This bounds the blast
  radius: the rewrite targets booking promotion, not the client enrichment plumbing.
- **KTD5: minimal `DOCS/SCORING.json`.** Delete `scoreWeights`, `gates`,
  `minimumForLeedReady` / `minimumForOutreachReady`, and the dead `verification` block.
  Keep `hotDaysOut`, a new `futureMinHours` (default 12), `requiredFields` (the addLeed
  contract), and `defaults` (pr 0, sh '*'). The startup loader and derived constants
  (:665-678) shrink to match.

## Implementation Units

### U1. Shrink DOCS/SCORING.json and the startup loader

**Goal:** Retire the points/gate policy to the minimal knob set; keep the engine
booting against the new shape.

**Requirements:** KTD5; tuning-surface requirement.

**Files:** `DOCS/SCORING.json`, `server/mcp/mcp_server.js` (loader + derived constants
~:661-690).

**Approach:** Delete `booking.scoreWeights`, `booking.gates`,
`minimumForLeedReady`/`minimumForOutreachReady`, `factlet.points*`, and the
`verification` block from `DOCS/SCORING.json`. Keep/add `hotDaysOut`,
`futureMinHours: 12`, `requiredFields: ["tn","ti","lc","zp","st","em","dt"]`,
`defaults: { pr: 0, sh: "*" }`, and `genericEmailPrefixes`. Remove the now-dead
constants (`SIGNAL_POINTS`, `FACTLET_THRESHOLD`, `FACTLET_POINTS_PER`,
`DRAFT_THRESHOLD_CLIENT`) and the `evaluateGate` function (:690) once U2 stops calling
it. Keep `GENERIC_EMAIL_PREFIXES`.

**Patterns to follow:** Existing `SCORING` load + `JSON.parse` at :665-674.

**Test scenarios:**
- Server boots with the trimmed `DOCS/SCORING.json` and no reference error from a
  removed constant.
- A `DOCS/SCORING.json` missing `requiredFields` fails fast at startup with a clear
  message (mirror the existing FATAL-on-malformed behavior at :670).

**Verification:** `mcp_server.js` starts; `grep` shows no remaining reads of the
deleted keys/constants.

### U2. Replace the booking classifier with cold/brewing/hot

**Goal:** Rewrite `computeBookingTargetScore` so it sets `Booking.status` to
cold/brewing/hot and drops `bookingScore` and the gate logic.

**Requirements:** KTD1, KTD2, KTD3; states + ordering requirements.

**Dependencies:** U1, U3 (the judge), U4 (the procedural floor helper).

**Files:** `server/mcp/mcp_server.js` (`computeBookingTargetScore` ~:1075-1164 and its
callers).

**Approach:** Rename to `classifyBooking`. Procedural pre-pass: if startDate is absent
or not at least `futureMinHours` in the future, the leed cannot be hot (cap at brewing
or cold per enrichment state). Otherwise call the judge (U3) with the synthesized
dossier + booking + active bar, and write the returned `cold | brewing | hot` to
`status`. Remove `bookingScore`, the `leedReady`/`outreachReady` gate calls, the
`hot`-flag derivation, and `shareReady`. Acting on a leed (share or outreach drafted)
sets `status` back to `cold`.

**Test scenarios:**
- Booking with a future start + dossier the judge calls hot -> `status = "hot"`.
- Booking with a past start -> never hot (brewing or cold), regardless of dossier.
- Booking under active enrichment, judge returns brewing -> `status = "brewing"`.
- A freshly identified client/booking with no enrichment -> `status = "cold"`.
- After a share or outreach action, the booking's `status` returns to `cold`.

**Verification:** Rerunning the flow on regenerated data yields bookings distributed
across cold/brewing/hot with no `leed_ready`/`outreach_ready` values remaining.

### U3. The mode-aware LLM judge

**Goal:** One LLM call that reads the dossier + booking and returns cold/brewing/hot
against the active bar, replacing `detectDemandSignal` + `_llmJudgeDemand`.

**Requirements:** KTD2; demand (explicit or inferred-at-high-confidence), PMF,
decision-maker, opportunity-real.

**Dependencies:** U1.

**Files:** `server/mcp/mcp_server.js` (`detectDemandSignal` :975, `_llmJudgeDemand`
:931).

**Approach:** Replace both with `judgeLeed(dossier, booking, mode)`. The prompt encodes
CLASSIFICATION.md: real client, decision-maker for VALUE_PROP, opportunity real even if
unrequested, product-market fit, and demand (explicit request, or inferred at high
confidence from prior hiring / PMF / reports from similar providers in the dossier).
`mode` selects the bar: outreach (lower) or marketplace (higher, demand must be high
confidence). Return one of `cold | brewing | hot` plus a short reason appended to the
dossier prose (no new field). Reuse the existing LLM config (`Config.llm*`) and call
pattern. Runs only on dossier-change (the existing enrichment/save path that currently
triggers scoring).

**Patterns to follow:** Existing LLM call shape in `_llmJudgeDemand` (:931), provider
handling, and `Config.llm*` fields.

**Test scenarios:**
- Explicit request in the booking text -> hot in either mode.
- Prom-pattern dossier (decision-maker + future themed event + PMF, no explicit
  request) -> hot in marketplace mode when inferred demand is strong; otherwise hot in
  outreach mode but brewing in marketplace mode.
- Thin dossier (no decision-maker or no PMF) -> brewing.
- Dossier showing the opportunity is dead (event passed, slot filled, publicly stated)
  -> not hot; the judge demotes.
- Judge is invoked on dossier-change only, not on every read (assert call count across
  repeated reads with no dossier change).

**Verification:** Same dossier + mode produces a stable state across repeated runs;
changing mode can change the state for a borderline inferred-demand leed.

### U4. Procedural addLeed-composability floor

**Goal:** A pure helper that answers "can a complete addLeed payload be composed from
this dossier?" used to gate the share action.

**Requirements:** KTD3; marketplace floor (full contract + future start).

**Dependencies:** U1.

**Files:** `server/mcp/mcp_server.js` (new helper near the share path; `pipelineShareBooking`
share path ~:1266 / share_booking handler).

**Approach:** `canComposeAddLeed(client, booking)` returns `{ ok, missing: [...] }` by
checking presence of `requiredFields` (tn, ti, lc, zp, st, em, dt) and start date at
least `futureMinHours` in the future. `et` and `ph` are not required. The share action
calls it and refuses with the missing list if not ok; addLeed remains the final judge
of location precision. Outreach has no addLeed floor: it needs only a real client name
+ a real direct email (reuse the existing direct-email / `GENERIC_EMAIL_PREFIXES`
check).

**Test scenarios:**
- All required fields present + future start -> ok, empty missing list.
- Missing zip -> not ok, `missing` contains `zp`.
- Past start date -> not ok, `missing` contains the date.
- Missing `et` / `ph` -> still ok (optional).
- Generic inbox email (info@, events@) -> fails the outreach direct-email check.

**Verification:** Sharing an incomplete hot leed is refused with the missing-field
list; sharing a complete one proceeds to the addLeed call.

### U5. Delete live-URL re-verification; capture proof once

**Goal:** Remove the save/share live-fetch that blocks promotion; trust evidence
captured at fetch.

**Requirements:** Scope decision (include the Catch-22 fix); origin bug doc.

**Files:** `server/mcp/mcp_server.js` (`verifyEvidenceUrl` :1862,
`verifyResolvedDateSource` :1489, `validatePatchEvidenceUrls` :1888, the factlet-save
path ~:2808).

**Approach:** Remove the live-fetch calls from the save and share paths. When a worker
first reads a page, store the captured snippet + date-seen on the factlet (an existing
factlet field or the factlet content prose); trust it afterward. No re-fetch on save or
share. Keep any non-network validation that does not hit the source URL.

**Execution note:** Characterize the current save path with a quick test before removing,
so the removal does not silently drop a still-needed non-network check.

**Test scenarios:**
- Saving a booking/factlet whose sourceUrl is an RFP/PDF/aggregator page succeeds (no
  live fetch, no rejection).
- The captured snippet + date-seen persist on the factlet at fetch time.
- No code path performs an HTTP fetch of `sourceUrl` during save or share.

**Verification:** A booking sourced from a page that does not literally restate the date
can still reach hot and be shared.

### U6. Adapt rescore, status counts, share_booking, and mode wiring

**Goal:** Make the rest of the engine speak cold/brewing/hot and route hot leedz by mode.

**Requirements:** Ordering (soonest event first); interactive vs headless action.

**Dependencies:** U2, U3, U4.

**Files:** `server/mcp/mcp_server.js` (`rescore` :2291-2325, status counts
:2350-2480, `pipelineShareBooking` :1266, the SHOW_HOT_LEEDZ path, the
EMAIL_SHARE_PATHS auto-mirror :3035), `DOCS/FOUNDATION.md`.

**Approach:** `rescore` re-runs `classifyBooking` over non-terminal bookings. Status
counts report cold/brewing/hot instead of the old enum. Headless action reads
`Config.defaultBookingAction`: marketplace shares hot leedz that pass
`canComposeAddLeed`; outreach drafts emails for hot leedz with a real contact.
Interactive: SHOW_HOT_LEEDZ presents hot leedz ordered by soonest startDate; the user
picks share or outreach per leed. Edit `DOCS/FOUNDATION.md` to drop the "demand is
never stored" / numeric-score language and point to `DOCS/CLASSIFICATION.md`.

**Test scenarios:**
- `rescore` over a mixed set leaves only cold/brewing/hot values.
- Status report returns cold/brewing/hot counts; no `leed_ready` key.
- SHOW_HOT_LEEDZ returns hot leedz ordered by soonest startDate ascending.
- Headless marketplace mode shares only hot leedz that pass `canComposeAddLeed`;
  outreach mode drafts emails and does not attempt a share.

**Verification:** A full flow run in each headless mode acts only on hot leedz and via
the correct action; interactive presents the hot list.

## Scope Boundaries

In scope: the booking classification + promotion rewrite, the LLM judge, the procedural
share floor, the live-URL deletion, and the minimal SCORING.json, all in the PRECRIME
source tree.

Out of scope:
- The Leedz Lambdas (`addLeed.py`, `agent_shareLeed`) — already done; PRECRIME stays a
  separate product with no shared code.
- `Client.dossierScore` removal — kept as an internal enrichment signal (KTD4).
- Thread 1 (the procedural conductor / parallelization) — independent workstream.

### Deferred to Follow-Up Work
- Any client-side cold/brewing/hot surfacing beyond what booking classification needs.
- Pricing UX for interactive marketplace shares beyond the pr 0 default.

## Risks & Dependencies

- **LLM determinism.** The judge can return different states for a borderline dossier
  across runs. Mitigation: a tight, rubric-anchored prompt; the share floor (U4) and
  addLeed are deterministic backstops, so a wrong "hot" cannot post an incomplete leed.
- **Data regeneration.** Old rows carry retired status values. Mitigation: regenerate by
  rerunning the flow (Thread 3); `rescore` reclassifies any survivors.
- **Removing live-URL verify could drop a still-needed non-network check.** Mitigation:
  the U5 execution note characterizes the save path first.
- **Test harness unknown.** PRECRIME has no obvious unit-test runner in `server/mcp`.
  The test scenarios above are written as behaviors; confirm or stand up a lightweight
  runner at execution time, else verify via `rescore` + a flow run on regenerated data.
