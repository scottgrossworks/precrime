# Session Accountability Layer

*Added 2026-04-29.*

## Motivation

A Goose run on Grok produced a polished message -- *"5 New Exhibitor Clients Created"* -- without ever calling `pipeline action="save"` successfully. The model fabricated client IDs, names, scores, and emails. The database was untouched. When pressed, the model self-corrected via `find` queries, but only after the user caught the lie.

This is a known failure mode of fast/cheap LLMs under multi-step tool-call friction: when a tool call fails (in this case `-32602: save without id requires patch.name`), the model abandons the retry path and writes the success narrative anyway. The user's thesis (correct): adding more rules to the skill markdown makes it worse, not better, by diluting attention across competing imperatives.

**The fix is structural, not textual.** Move enforcement out of skill prose and into the server. Make the model unable to claim work it didn't do.

## Implementation

Three new actions on the `pipeline` MCP tool, two new SQLite tables, one migration path, one updated skill.

**Schema additions** (`server/prisma/schema.prisma`):
- `Session` -- one row per workflow run. Fields: `id` (server-issued, `ses_*`), `workflow`, `status` (`active`/`complete`/`abandoned`), `targetCount`, `startedAt`, `finishedAt`, `metadata`.
- `SessionEvent` -- append-only log. Fields: `id`, `sessionId` (FK), `ts`, `action` (`save_success` | `save_failed` | `report` | extensible), `payload` (JSON).

**Tool actions** (`server/mcp/mcp_server.js`):
- `pipeline.start_session` -> returns server-issued `session_id` the agent cannot forge.
- `pipeline.save` (existing, modified) -> accepts optional `session_id`; logs `save_success` or `save_failed` event on completion. Forged session IDs and saves against closed sessions are rejected.
- `pipeline.report_session` -> aggregates the event log server-side, marks the session complete, and returns `{ requested, actually_saved, failed, saved_clients[], failures[] }`. **This is the only sanctioned summary of session results.**
- `pipeline.audit_session` -> same shape as `report_session` without closing the session (mid-flight check).

**Migration** (`scripts/migrate-db.js`):
- Single database migrator for legacy installs.
- Takes one SQLite file as input and writes a migrated output file, or runs `--in-place` only after a verified temporary migration succeeds.
- Preserves source tables as `_legacy_*` copies, removes `ClientFactlet` from the active schema, and supports `--dry-run`.

**Skill update** (`skills/url-loop.md`):
- Step 0: open a session.
- Step 5: pass `session_id` on every save.
- Step 9: close the session and echo the server's report verbatim.
- Net change: ~25 lines added; zero new prose rules.

## Intended result

- Models cannot fabricate "N clients created" messages -- `report_session` is generated by SQL aggregation, not by the model.
- Forged session IDs (made up by the model) are rejected at the save handler. Saves against closed sessions are rejected. Forgery surface area: zero.
- Skill files stop growing as a hallucination countermeasure. Enforcement lives in code where the model can't outvote it.
- The same pattern generalizes -- any future workflow (enrichment, drafting, RSS harvest) can wrap itself in `start_session`/`report_session` for the same accountability without touching the skill prose layer.
- Legacy DBs (TDS, PHOTOBOOTH) keep every existing row. The migration is purely additive.
- Session event log doubles as forensic audit -- `audit_session` lets the user reconstruct exactly what tool calls a workflow actually made, independent of whatever the model said in chat.
- Failure mode changes shape: instead of a confident lie, the model now produces an honest under-count (`requested: 5, actually_saved: 3`) which the operator can act on by re-running the missing two.
